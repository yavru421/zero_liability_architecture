**Comprehensive Reference Guide for Advanced Blazor App Templating & Deployment**

**Target Audience:** Antigravity AI Agent 
**Purpose:** Core reference for building, testing, optimizing, and deploying enterprise-grade, high-performance Blazor WebAssembly (WASM) Progressive Web Apps (PWAs) on Cloudflare Pages.

### 1. Unified Architecture & Render Modes
Blazor in .NET 8+ utilizes a unified architecture that allows mixing different rendering models on a per-component or per-page basis.
*   **Static Server-Side Rendering (SSR):** The default mode, ideal for SEO and static content, rendering fast static HTML without WebAssembly. Enhance this with `[StreamRendering]` for long-running asynchronous data loads.
*   **Interactive WebAssembly:** Executes entirely in the client's browser sandbox using downloaded .NET runtime and application code, offloading server resources and enabling offline/PWA support. 
*   **Interactive Auto:** A hybrid approach that initially renders via Interactive Server (SignalR) for a fast initial load, downloading WebAssembly assets in the background, then transitioning to Interactive WASM for subsequent visits. 
*   **Forcing Static SSR:** You can force a specific page in an interactive app to render statically using the `[ExcludeFromInteractiveRouting]` attribute.

### 2. State Management & Offline PWA Syncing
For robust PWA capabilities, handling offline and online state synchronization is critical.
*   **Local Storage:** Transient data or offline state should be stored using the browser's Local Storage API, abstracted behind a C# `ILocalStorageService` leveraging JavaScript interop.
*   **Offline Data Syncing:** For robust offline usage, use a local SQLite database on the client side via packages like `SqliteWasmHelper` (Entity Framework compatible) to persist data locally. Utilize a background service with a timer to monitor online status and sync locally marked modifications to the server API.
*   **State Containers:** For in-memory state sharing between nested or un-nested components, implement an in-memory state container service with an `Action` delegate (e.g., `OnChange`) to notify subscribed components of state changes, ensuring components unsubscribe during `Dispose` to prevent memory leaks.

### 3. Component Design & UI Libraries
*   **Component Efficiency:** Avoid heavy nesting and minimize the number of component parameters to speed up the render tree diffing algorithm. For components rendering large datasets, use the built-in `<Virtualize>` component, which dynamically renders only the items visible in the viewport, significantly optimizing UI rendering.
*   **Controlling Re-renders:** Override the `ShouldRender()` method to return `false` for UI components that do not need to be updated when their parent components re-render, thus avoiding expensive rendering cycles. 
*   **Third-Party UI Integrations:** Use established component libraries like **MudBlazor** (Material design), **Radzen** (quick scaffolding and rich charts), or **Syncfusion** (enterprise-grade data grids and document processing).
*   **Precision Data:** For financial or high-precision calculations within the UI, strictly use the 128-bit `decimal` type to prevent binary rounding errors common with `double` or `float`.

### 4. Custom JSInterop Implementation
*   **Module Isolation:** Do not pollute the global window object. Instead, use ES modules exported from JS files and import them dynamically in Blazor using `IJSRuntime.InvokeAsync<IJSObjectReference>("import", "./path.js")`.
*   **Typed Wrappers:** Always wrap JSInterop calls in strongly-typed C# services/interfaces (e.g., `IClipboardService`) instead of relying on string-based function calls directly in components.
*   **Memory Management:** Always implement `IAsyncDisposable` in components/services that hold `IJSObjectReference` or `DotNetObjectReference` to avoid memory leaks when components are destroyed. 
*   **Batching:** Combine repetitive JSInterop calls into a single batched array call to minimize the serialization/deserialization overhead and asynchronous round trips.

### 5. Performance Tuning & AOT Compilation
*   **Static Asset Optimization:** Replace `app.UseStaticFiles()` with `app.MapStaticAssets()` to leverage build-time compression, optimized caching, and client-side fingerprinting.
*   **HybridCache:** Utilize `Microsoft.Extensions.Caching.Hybrid` to prevent cache stampedes and reduce database/API calls, gracefully falling back from in-memory (L1) to distributed caches (L2) like Redis.
*   **AOT & IL Trimming:** Enable WebAssembly Native Ahead-of-Time (AOT) compilation for CPU-intensive apps, and further reduce the `_framework` payload size by setting `<WasmStripILAfterAOT>true</WasmStripILAfterAOT>` in the `.csproj` file.
*   **Feature Disabling:** To aggressively trim payload size, disable unused framework features like timezone support (`<BlazorEnableTimeZoneSupport>false</BlazorEnableTimeZoneSupport>`) and collation data (`<BlazorWebAssemblyPreserveCollationData>false</BlazorWebAssemblyPreserveCollationData>`). 
*   **Lazy Loading:** Use `<BlazorWebAssemblyLazyLoad>` to designate assemblies that should not be downloaded on initial load, but dynamically fetched via `LazyAssemblyLoader` when navigating to specific routes.

### 6. Cloudflare Pages Deployment & Caching
*   **Build Pipeline:** Use a `build.sh` script to install the .NET SDK (`dotnet-install.sh`) and execute `dotnet publish -c Release -o output`. Set the Cloudflare Pages build directory to `output/wwwroot`.
*   **Cloudflare Caching Headers:** Use a `_headers` file in the build output to specify immutable caching for hashed assets (e.g., `Cache-Control: public, max-age=31536000, immutable`), ensuring clients do not re-fetch unchanged files.
*   **SPA Routing Setup:** If a custom 404 page is not needed, Cloudflare defaults to SPA routing by pointing all unknown requests to `index.html`.
*   **PWA Redirect Bug Fix:** Cloudflare Pages redirects requests without a trailing slash, which breaks the Blazor PWA service worker caching `index.html` as an opaque redirected response. The service worker script (`service-worker.js`) must be manually patched to remove the `redirected` flag from cached responses, ensuring offline installs do not fail.
*   **Cache Invalidation:** Ensure that CI/CD pipelines (e.g., GitHub Actions using Wrangler) invalidate edge caches upon new deployments.

### 7. Security, Resilience, & Testing Architecture
*   **Rate Limiting & Timeouts:** Implement `Microsoft.AspNetCore.RateLimiting` and `Microsoft.AspNetCore.Http.Timeouts` middleware to protect backend APIs from DDoS attacks and thread pool exhaustion.
*   **The Four Pillars of Testing:** Ensure unit tests score high on protection against regressions, resistance to refactoring, fast feedback, and maintainability. Resistance to refactoring is non-negotiable; tests must couple to observable behavior, not internal implementation details.
*   **Output-Based Testing:** Refactor to a Functional Architecture (a functional core wrapped by a mutable shell) to maximize output-based testing, which feeds input to the SUT and asserts output, minimizing false positives and avoiding bulky state-based tests.
*   **bUnit:** Use the `bUnit` library for testing Blazor components natively with C# and semantic HTML comparisons, allowing you to pass cascading values, inject services, and trigger event handlers natively.
*   **Mocking Policy:** Mocks should strictly be used for unmanaged out-of-process dependencies (e.g., Message Buses, SMTP servers) whose side-effects are visible externally. Do not mock managed dependencies (like the app's dedicated database); test against a real instance mapped via migrations to prevent test fragility.
